# Implementation Report (Feb 21, 2026)

This report summarizes the recent work on `pystocks/session.py`, `pystocks/research_correlations.py`, and `pystocks/fundamentals.py`, the observed outcomes, and the recommended next steps.

## 1) Executive Summary

The fundamentals pipeline is now significantly more robust and data-driven than the prior static endpoint strategy:

- `mf_performance` and `mf_risks_stats` now use largest-to-smallest period fallback (`10Y -> 5Y -> 3Y -> 1Y -> 6M`).
- A landing-first dynamic fetch policy is implemented in `pystocks/fundamentals.py`.
- A research script (`pystocks/research_correlations.py`) was improved and run at scale (`sample_size=500`) to empirically estimate which landing teasers predict useful data in full endpoints.
- Low-yield endpoints are de-prioritized by default (ESG supplementary endpoint, ownership, sentiment tick), while high-yield endpoints are prioritized.

## 2) What Was Changed

### 2.1 `pystocks/research_correlations.py`

Implemented and validated:

- Landing-first feature extraction for teaser presence.
- Period fallback measurement for `perf` and `risk` with attempt traces.
- Robust status handling (non-200 and network issues do not crash the run).
- Output files:
  - `research_yields.csv` (raw per-conid feature/yield/status rows)
  - `research_correlations_summary.csv` (conditional probabilities and lift)

### 2.2 `pystocks/fundamentals.py`

Implemented and validated:

- Period fallback for:
  - `mf_performance/{conid}?risk_period={period}&statistic_period={period}`
  - `mf_risks_stats/{conid}?period={period}`
- Landing-first flow for dynamic endpoint selection.
- Morningstar fetch is now unconditional.
- `sma_search` (1Y historical window) added and gated by `landing.overall_ratings`.
- Low-yield endpoints kept opt-in via `FETCH_LOW_YIELD_ENDPOINTS`.
- Snapshot includes selected periods (`performance_period`, `risk_stats_period`) and probe metadata.

### 2.3 `pystocks/session.py`

Observed behavior:

- If `auth_state.json` is valid, browser may open and close quickly with no manual login.
- If auth is stale, manual login is required.
- Current script does not implement auto-reauth retries or a multi-stage fallback flow.

## 4) What Is Going Well

- The fallback strategy materially increases robustness for younger instruments with incomplete long-history periods.
- Landing-first gating reduces unnecessary calls for low-value endpoints.
- Core coverage is strong for profile/holdings/dividends and good for ratios/perf/risk/mstar.
- Saved snapshots now carry more diagnostic context (`probe` flags and selected periods).

## 5) What Is Going Poorly or Risky

- `impact/esg` supplementary endpoint currently has effectively no useful payload in this universe sample.
- `ownership` and `sma_tick` are very low-yield globally; they should remain off by default except targeted use.
- `sma_search` is better than tick/high_low but still low precision if called too broadly.
- Session validity is still brittle over time (no built-in automatic reauth flow).
- `research_correlations.py` and `fundamentals.py` duplicate some response-heuristic logic and could diverge if edited independently.

## 6) Recommended Next Steps (Priority Order)

1. **Stabilize auth loop in `session.py`**
- Add a reauth trigger path when fundamentals receives 401/403.
- Save state after successful dashboard load with fallback selectors.

2. **Finalize a stable hardcoded fetch policy**
- Keep always-on: profile, holdings, dividends, morningstar.
- Keep conditional: ratios/perf/risk/lipper/search.
- Keep off by default: impact/esg, ownership, sma_tick/high_low.

3. **Run one larger validation pass with current policy**
- Capture request volume and useful-payload volume per endpoint.
- Compare against previous policy to quantify call reduction and data loss.

4. **Parser/ETL implementation**
- Build flattening logic for variable schemas into analysis-friendly tabular form.
- Persist parser diagnostics (missing keys, schema variants, endpoint status mix).

Use this order:

- `research_yields.csv`
- `research_correlations_summary.csv`
