# Amended Plan: Consistent Dual-Layer Storage (Raw CAS + Parquet Analytics) with Time-Series-Aware Handling

## Summary
1. Keep one consistent storage architecture for all endpoints: exact raw payloads in CAS blobs plus analytics-ready parquet datasets.
2. For complex endpoints (`ownership`, `dividends`, later `sentiment_search`), preserve full raw payload in CAS, but write curated analytics parquet outputs that exclude unnecessary embedded price series.
3. Keep short-selling ingestion deferred.
4. Continue cleanup track after this storage refactor: remove `risk_stats`, remove tick/high-low sentiment snapshots, and promote `sentiment_search` to normalized series handling.

## High-Level Architecture (Final)
1. Raw layer (unchanged in principle):
- Store exact HTTP JSON response in CAS blob for every fetched endpoint.
- Keep lineage in `/Users/alex/Documents/pystocks/data/fundamentals/events.db` (`endpoint_events`, `blobs`).
2. Analytics layer (parquet):
- Store endpoint analytics rows for all endpoints.
- For complex endpoints, store canonical endpoint snapshot rows plus dedicated row-level series tables.
- All analytics rows include lineage fields to raw payload (`payload_hash`, `endpoint_event_id`, `conid`, `observed_at`, `effective_at`).
3. Query layer (DuckDB views):
- Keep existing endpoint views.
- Add views for normalized complex datasets and series datasets.

## Storage/Lineage and Dedupe Rules
1. CAS dedupe remains by `payload_hash` as today.
2. Endpoint event dedupe remains `(conid, endpoint, effective_at, payload_hash)`.
3. Add analytics-row dedupe manifest table in `events.db`:
- table: `analytics_rows`
- unique key: `(analytics_name, row_key, payload_hash)`
4. `row_key` composition:
- `dividends_events`: hash of (`conid`, `event_date`, `amount`, `description`, `payload_hash`)
- `ownership_trade_log`: hash of (`conid`, `trade_date`, `party`, `action`, `shares`, `value`, `payload_hash`)
5. Every analytics row must reference `payload_hash` and `endpoint_event_id`.

## Implementation Steps

### Phase 1: Refactor store pipeline for dual-layer consistency
1. Keep scraper fetch payloads raw.
2. Persist raw payload to CAS + endpoint event index first.
3. Dispatch endpoint-specific analytics normalizers after raw persistence.
4. For non-complex endpoints, keep existing flattening behavior.
5. For `dividends` and `ownership`, replace raw-flatten endpoint parquet with canonical snapshot writers + dedicated analytics dataset writers.

### Phase 2: Build complex endpoint normalizers
1. Add normalization module (new file):
- `/Users/alex/Documents/pystocks/pystocks/fundamentals_normalizers.py`
2. Implement:
- `normalize_dividends_snapshot(payload)`
- `extract_dividends_events(payload)`
- `extract_dividends_industry_metrics(payload)`
- `normalize_ownership_snapshot(payload)`
- `extract_ownership_trade_log(payload, drop_no_change=True)`
3. Ensure embedded price series are ignored for analytics extraction.

### Phase 3: Immediate backfill migration
1. Iterate historical `endpoint_events` for `dividends` and `ownership`.
2. Rehydrate raw payloads from CAS blobs.
3. Recompute canonical endpoint parquet rows and analytics datasets.
4. Rewrite affected parquet partitions for these endpoints/datasets.
5. Refresh DuckDB views and dataset catalog.
6. Keep CAS blobs untouched.

### Phase 4: Post-foundation cleanup track
1. Remove `risk_stats` fetch and analytics wiring.
2. Remove tick/high-low sentiment snapshot fetch/store wiring.
3. Update `sentiment_search` to long-range daily fetch policy and bar-level analytics dataset.
