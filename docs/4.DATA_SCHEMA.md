# Data Schema Documentation

The data layer uses a hybrid model:

- SQLite for scraper run tracking and endpoint call logs (`data/pystocks.db`).
- CAS + Parquet + DuckDB for fundamentals payload storage and time-series analytics (`data/fundamentals/`).

## 1) Instrument Universe (`data/ib_products.csv`)

Seed universe from IBKR product search.

| Column | Description |
| --- | --- |
| `symbol` | Ticker symbol |
| `exchangeId` | Primary exchange |
| `conid` | IBKR instrument ID (primary key) |
| `isin` | ISIN |
| `currency` | Trading currency |

## 2) SQLite Registry (`data/pystocks.db`)

Tracks scraper progress and endpoint-level HTTP logs.

### Table: `instruments`

| Column | Description |
| --- | --- |
| `conid` | Primary key |
| `symbol` | Ticker |
| `exchange` | Exchange |
| `isin` | ISIN |
| `last_scraped_fundamentals` | Last successful fundamentals date (`YYYY-MM-DD`) |
| `last_status_fundamentals` | Status label (`success`, `error_401`, etc.) |

### Table: `scraper_logs`

| Column | Description |
| --- | --- |
| `conid` | Instrument ID |
| `timestamp` | API call timestamp |
| `endpoint` | Endpoint path called |
| `status_code` | HTTP status |
| `error_message` | Error details (if any) |

## 3) Fundamentals Storage (`data/fundamentals/`)

### 3.1 CAS blobs (`data/fundamentals/blobs/`)

- Content-addressed compressed payload blobs (`sha256` hash key).
- Format: Zstandard-compressed canonical JSON (`*.json.zst`).
- Purpose: lossless raw payload retention with deduplication.

### 3.2 Event index (`data/fundamentals/events.db`)

SQLite manifest that tracks:

- endpoint event identity (`conid`, `endpoint`, `effective_at`, `payload_hash`)
- observed timestamp (`scraped_at`/observed time)
- effective date and source (endpoint as-of or fallback policy)
- payload hash and blob path
- payload raw/compressed size

`UNIQUE(conid, endpoint, effective_at, payload_hash)` prevents duplicate event inserts.

### 3.3 Endpoint-partitioned Parquet (`data/fundamentals/parquet/`)

Partition layout:

- `endpoint=<endpoint_slug>/year=<YYYY>/month=<MM>/*.parquet`

Each row represents one endpoint event for one instrument/effective date and contains:

- fixed metadata columns (conid, endpoint, observed/effective dates, hash, blob path, sizes)
- dynamically flattened payload scalar fields (`f__...`)
- list-length and scalar-list JSON helper fields for variable payload sections

Missing fields are naturally `NULL`, enabling schema growth over time.

### 3.4 DuckDB query layer (`data/fundamentals/fundamentals.duckdb`)

Generated views:

- `endpoint_events_all` (union across all endpoint partitions)
- `endpoint_<endpoint_slug>` per endpoint
- `endpoint_catalog` summary view

This supports SQL analytics without copying Parquet into a database file.

## 4) Effective Date Policy

Effective dates are not tied to scrape day. They are resolved as:

1. endpoint `as_of_date`/`asOfDate` for discrete endpoints (`ratios`, `holdings`, etc.)
2. fallback to `ratios.as_of_date` when endpoint-level date is unavailable
3. fallback to earliest discrete endpoint date in the same scrape
4. fallback to observed scrape date

This yields irregular, endpoint-specific discrete time series.

## 5) Research Outputs (Policy Tuning)

The correlation research utility writes:

- `data/research/research_yields.csv`
  - Per-conid landing features, endpoint yields, status codes, selected periods.
- `data/research/research_correlations_summary.csv`
  - Conditional probabilities and lift values (`P(target|feature)` and `P(target|not feature)`).

These files are used for manual policy calibration and are not required for runtime scraping.

## 6) Fundamentals Run Telemetry

Produced by `pystocks/fundamentals.py` each run:

- `data/research/fundamentals_run_telemetry_<timestamp>.json`
- `data/research/fundamentals_run_telemetry_latest.json`

Contents include:

- Per-endpoint call counts.
- Per-endpoint useful payload counts/rates.
- Status-code distribution by endpoint family.
- Run summary (`targeted`, `processed`, `saved`, `inserted_events`, `duplicate_events`, `auth_retries`, `aborted`).

## 7) Validation/Compare Outputs

- Policy-run comparison:
  - `data/research/fundamentals_telemetry_compare_<timestamp>.csv`
