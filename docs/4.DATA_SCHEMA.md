# Data Schema Documentation

The data layer uses a hybrid SQLite/JSON architecture to manage the instrument universe and historical fundamental snapshots.

## 1. Instrument Universe (`data/ib_products.csv`)
This CSV acts as the initial "seed" for the universe, scraped from the IBKR product search API.
| Column | Description |
| --- | --- |
| symbol | The ticker symbol. |
| exchangeId | Primary exchange code. |
| conid | **Primary Key.** IBKR internal unique identifier. |
| isin | International Securities Identification Number. |
| currency | Trading currency. |

## 2. SQLite Registry (`data/pystocks.db`)
Tracks metadata, scraping status, and logs for all instruments.

### Table: `instruments`
| Column | Description |
| --- | --- |
| conid | **Primary Key.** IBKR internal unique identifier. |
| symbol | Ticker symbol. |
| exchange | Exchange ID. |
| isin | ISIN identifier. |
| last_scraped_fundamentals | Date (YYYY-MM-DD) of the last successful scrape. |
| last_status_fundamentals | Status of the last scrape (`success`, `error_401`, etc.). |

### Table: `scraper_logs`
| Column | Description |
| --- | --- |
| conid | Associated instrument ID. |
| timestamp | Time of the API call. |
| endpoint | The specific API endpoint called. |
| status_code | HTTP status code (e.g., 200, 401). |
| error_message | Detailed error if the call failed. |

## 3. Fundamental Snapshots (`data/fundamentals/`)
Historical data is stored as raw JSON blobs to preserve Point-in-Time (PIT) integrity.
*   **Path:** `data/fundamentals/{conid}/{YYYY-MM-DD}.json`
*   **Content:** A combined JSON object containing full responses from the `landing`, `mf_profile_and_fees`, and `mf_holdings` endpoints.

## 4. TimeSeries Directory (`data/daily-trades/series/*.csv`)
*   **Format:** `SYMBOL-EXCHANGE-CURRENCY.csv`.
*   **Goal:** Standardize historical price series for regression analysis.
