# Data Schema Documentation

The data layer uses a hybrid SQLite + JSON architecture.

## 1) Instrument Universe (`data/ib_products.csv`)

Seed universe from IBKR product search.

| Column | Description |
| --- | --- |
| `symbol` | Ticker symbol |
| `exchangeId` | Primary exchange |
| `conid` | IBKR instrument ID (primary key) |
| `isin` | ISIN |
| `currency` | Trading currency |

## 2) SQLite Registry (`data/pystocks.db`)

Tracks scrape progress and endpoint-level logs.

### Table: `instruments`

| Column | Description |
| --- | --- |
| `conid` | Primary key |
| `symbol` | Ticker |
| `exchange` | Exchange |
| `isin` | ISIN |
| `last_scraped_fundamentals` | Last successful fundamentals date (`YYYY-MM-DD`) |
| `last_status_fundamentals` | Status label (`success`, `error_401`, etc.) |

### Table: `scraper_logs`

| Column | Description |
| --- | --- |
| `conid` | Instrument ID |
| `timestamp` | API call timestamp |
| `endpoint` | Endpoint path called |
| `status_code` | HTTP status |
| `error_message` | Error details (if any) |

## 3) Fundamental Snapshots (`data/fundamentals/{conid}/{YYYY-MM-DD}.json`)

Raw PIT snapshot containing the combined endpoint payloads for one run.
Snapshots are currently written as compact (minified) JSON to reduce storage footprint while preserving full dynamic response fidelity.

### Stable top-level keys (current expected set)

- `conid`
- `scraped_at`
- `landing`
- `profile_and_fees`
- `holdings`
- `ratios`
- `lipper_ratings`
- `dividends`
- `morningstar`
- `performance`
- `performance_period`
- `risk_stats`
- `risk_stats_period`
- `probe`

### Optional keys (policy-dependent)

- `sentiment_search`
- `ownership`
- `sentiment` (tick)
- `esg` (supplementary impact endpoint)

### Notes

- Endpoints can be omitted when dynamic routing decides not to fetch, or when fetched payload is structurally empty.
- `performance_period` and `risk_stats_period` reflect the successful period selected by fallback logic.
- `probe` contains landing-derived booleans used to route follow-up calls.

## 4) Research Outputs (Policy Tuning)

The correlation research utility writes:

- `data/research/research_yields.csv`
  - Per-conid landing features, endpoint yields, status codes, selected periods.
- `data/research/research_correlations_summary.csv`
  - Conditional probabilities and lift values (`P(target|feature)` and `P(target|not feature)`).

These files are used for manual policy calibration and are not required for runtime scraping.

## 5) Time Series Directory (`data/daily-trades/series/*.csv`)

- Filename pattern: `SYMBOL-EXCHANGE-CURRENCY.csv`
- Purpose: normalized historical prices for downstream factor/regression alignment.

## 6) Storage Direction

- Keep raw JSON snapshots as source-of-truth for heterogeneous IBKR payloads and PIT auditability.
- Build ETL/flattened analysis tables as a separate layer (`pystocks/preprocess.py` + future schema tables), rather than replacing raw snapshots.
- Evaluate archival compression/columnar storage after full-universe runs to quantify query/storage tradeoffs.
