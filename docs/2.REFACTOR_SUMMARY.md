# Fundamental Data Pipeline Refactor (Feb 2026)

## Overview
The PyStocks data acquisition layer has been refactored from a TWS-dependent (OCR/`ib_async`) model to a high-speed, direct API interception model using the IBKR Portal Proxy. This transition enables the collection of rich, structured JSON data for the entire ETF universe.

## Current Architecture

### 1. Data Intake (The "What")
The pipeline now targets three primary JSON endpoints per `conId`:
*   **`landing`**: Aggregates ratings (Lipper/Mstar), key ratios, and performance overview.
*   **`mf_profile_and_fees`**: Extracts expense ratios, management fees, and annual/prospectus report summaries.
*   **`mf_holdings`**: Captures granular sector (industry) and geographic (investor country) exposure.

### 2. Storage Strategy (The "Where")
*   **Registry (`data/pystocks.db`)**: A SQLite database manages instrument metadata and scraping logs. It tracks `last_scraped_fundamentals` to support incremental updates and skip-logic.
*   **Snapshots (`data/fundamentals/{conid}/{date}.json`)**: Discrete JSON files preserve the exact API response for every scrape date. This is the foundation for Point-in-Time (PIT) analysis.

### 3. Execution (The "How")
*   **`pystocks/session.py`**: Manages Playwright-based authentication and exports `httpx` cookies.
*   **`pystocks/fundamentals.py`**: Orchestrates the multi-endpoint fetch and manages rate limiting/concurrency.
*   **`pystocks/database.py`**: Handles instrument synchronization and logging.

## Technical Audit: Strengths & Weaknesses

### What it does WELL
*   **Data Density**: Captures significantly more fields than the previous OCR/TWS approach (e.g., Lipper ratings, detailed fee breakdowns, multiple benchmark comparisons).
*   **Reliability**: JSON responses are structured and consistent compared to fragile UI-based scraping.
*   **Incremental Progress**: The SQLite-backed skip-logic ensures that large-scale scrapes (17k+ instruments) can be resumed across multiple sessions without redundant API calls.
*   **PIT Integrity**: By saving raw JSON by date, the system avoids "Look-ahead Bias" in future backtests.

### What it does POORLY (or is missing)
*   **Authentication Fragility**: The `auth_state.json` expires unpredictably (usually < 24h). The system currently stops on 401/403 errors but cannot yet auto-reauthenticate without user intervention.
*   **Schema Volatility**: As noted in `@docs/5.ibkr_json_examples.md`, response keys vary by instrument (e.g., some have `mstar` style boxes, others don't). The "Raw JSON" storage handles this, but the downstream "Parser" needs to be highly defensive.
*   **Lack of Concurrency Control**: While using `asyncio`, the scraper processes instruments sequentially to avoid IP bans/rate-limiting. It does not yet implement a sophisticated "Token Bucket" or proxy rotation strategy.
*   **No "Change Detection"**: It currently scrapes the full snapshot daily. For metrics that change slowly (e.g., Domicile), this is inefficient.

## Amendments to Existing Documentation

### `docs/ANALYSIS_PIPELINE.md`
*   **Update**: Factor Construction can now include **Analyst Sentiment Factors** (using Lipper/Mstar ratings) and **Fee-Based Factors** (TER/Management Fee spreads).
*   **Update**: Data Alignment will now pull from the `data/fundamentals/` JSON tree rather than the legacy `fundamentals_*.csv`.

### `docs/DATA_SCHEMA.md`
*   **Amendment**: The "Proposed Schema Transition" is partially implemented. Instead of a single `fundamentals_archive.parquet` (which requires `pyarrow`), we are using a **FileSystem-as-a-DB** approach (nested JSONs). 
*   **Suggestion**: Once `pyarrow` is available, a "Compactor" script should be written to move these JSONs into a partitioned Parquet for performance.

### `docs/IBKR_PORTAL_API.md`
*   **Amendment**: The `landing` endpoint is currently the primary "Bulk" fetcher, but `mf_holdings` and `mf_profile_and_fees` are essential *supplements* because `landing` often returns truncated or "Top 10" only versions of those specific sections. `@pystocks/fundamentals.py` will need to be expanded to include the other endpoints described in `@docs/5.ibkr_json_examples.md`
