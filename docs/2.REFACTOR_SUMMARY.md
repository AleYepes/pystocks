# Fundamental Data Pipeline Refactor Summary (Feb 2026)

## Overview

The fundamentals pipeline is now API-driven and empirically tuned.

Compared with the earlier static endpoint strategy, the current version:

- fetches `landing` first,
- routes follow-up endpoint calls from landing teaser presence,
- uses period fallback for historical performance/risk endpoints,
- and stores endpoint payloads in CAS + Parquet event storage for auditability and time-series analysis.

## Current Fundamentals Execution Model

### Step 1: Session bootstrap

- `pystocks/session.py` loads existing `auth_state.json` when present.
- If auth is still valid, login may complete without manual interaction.

### Step 2: Landing-first probe

- `pystocks/fundamentals.py` calls `landing/{conid}?widgets=...` first.
- Teaser sections are used to choose which supplementary endpoints are worth calling.

### Step 3: Conditional endpoint fetch

High-priority endpoints:

- Always fetched: `mf_profile_and_fees`, `mf_holdings`, `dividends`, `mstar/fund/detail`
- Conditionally fetched: `mf_ratios_fundamentals`, `mf_lip_ratings`, `mf_performance`, `mf_risks_stats`, `sma/request?type=search`
- Opt-in only (low yield in current universe sample): `impact/esg`, `ownership`, `sma/request?type=tick`

### Step 4: Period fallback

- `mf_performance`: `10Y -> 5Y -> 3Y -> 1Y -> 6M`
- `mf_risks_stats`: `10Y -> 5Y -> 3Y -> 1Y -> 6M`

First successful payload wins and selected period is saved in endpoint event metadata.

## Empirical Results Driving the Policy

Research run: `pystocks/research_correlations.py --sample_size=500`

Observed useful-payload yields:

- `profile`: 100.0%
- `holdings`: 100.0%
- `dividends`: 100.0%
- `mstar`: 87.6%
- `ratios`: 82.4%
- `perf`: 83.8%
- `risk`: 80.0%
- `lipper`: 54.8%
- `sma_search`: 19.6%
- `sma_high_low`: 13.4%
- `sma_tick`: 4.4%
- `owner`: 2.8%
- `impact/esg`: 0.0%

Period fallback outcomes:

- `perf`: mostly `10Y`, with fallback/none for younger or sparse products.
- `risk`: more frequent fallback (`5Y/3Y/1Y/6M`) and some none.

## Strengths

- High structural reliability versus OCR/TWS scraping.
- Data-driven fetch policy instead of intuition-only endpoint selection.
- Lower waste on consistently low-yield endpoints.
- Endpoint-level PIT fidelity remains intact via immutable blob hashes + effective dates.

## Weaknesses / Remaining Gaps

- Auth still expires unpredictably; reauth handoff is implemented but still requires manual login interaction.
- Schema variability remains high; ETL layer is now present but should be expanded to include richer feature extraction.
- Research and production heuristics are duplicated and can drift.
- Endpoint policy is hardcoded and should be periodically recalibrated.

## Recommended Next Steps

1. Add automatic reauth handoff on 401/403.
2. Build parser/ETL (`pystocks/preprocess.py`) with schema-variant handling.
3. Run periodic correlation refreshes (e.g., weekly samples) and manually update hardcoded policy.
4. Add run-level telemetry: per-endpoint call count, useful payload count, and status code distribution.
